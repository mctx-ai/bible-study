## What This Is

A **Bible study MCP server** built with `@mctx-ai/app` framework, deployed on the mctx.ai platform at `bible-study.mctx.ai`.

**Purpose:** Deep Bible study for MCP-compatible AI clients. 155,510 verses across 5 public domain translations (KJV, WEB, ASV, YLT, Darby). Semantic search via 155K+ vector embeddings. 606,140 cross-references. 17,543 Strong's concordance entries with Hebrew and Greek lexicon definitions. 447,734 morphology records. 5,319 Nave's Topical Bible categories. Full-text search via FTS5.

**Key Features:**
- 7 tools: full-text search, semantic search, compare translations, cross-references, concordance, word study, topical search (with multi-level thematic retrieval)
- 3 resources: translations list, verse URI, chapter URI
- Cloudflare D1 (SQLite) for structured data, Vectorize for embeddings, Workers AI for embedding generation
- Comprehensive test coverage

## Architecture

### High-Level Structure

```
src/
  index.ts            ŌåÆ Server entry point (createServer, tool/resource registration)
  index.test.ts       ŌåÆ Comprehensive tests for all capabilities
  integration.test.ts ŌåÆ Integration tests that call live Cloudflare APIs (requires credentials)
  lib/
    bible-utils.ts      ŌåÆ Shared Bible utilities (book lookup, verse formatting)
    bible-utils.test.ts ŌåÆ Unit tests for bible-utils
    cloudflare.ts       ŌåÆ Workers-compatible Cloudflare API clients (D1, Vectorize, Workers AI)
    cloudflare.test.ts  ŌåÆ Unit tests for cloudflare client
    cloudflare-etl.ts   ŌåÆ Node-only ETL utilities (d1.batch, d1.batchFile, sqlLiteral)
  tools/              ŌåÆ One file per tool
    search-bible.ts   ŌåÆ Semantic vector search across verses (tool: semantic_search)
    find-text.ts      ŌåÆ Full-text keyword search via FTS5 (tool: find_text)
    concordance.ts    ŌåÆ Word/phrase frequency grouped by book (tool: concordance)
    compare-translations.ts ŌåÆ Side-by-side translation comparison
    cross-references.ts ŌåÆ Cross-reference lookup
    word-study.ts     ŌåÆ Original-language lexicon and morphology
    topical-search.ts ŌåÆ Multi-level thematic retrieval: verse-level (Nave's + Vectorize), major witnesses (book/narrative-level via Nave's topic expansion and book-density aggregation), match explanations
  resources/          ŌåÆ One file per resource (translations, verse, chapter)
scripts/              ŌåÆ ETL pipeline scripts (run locally with Node.js, NOT bundled)
dist/index.js         ŌåÆ Bundled output (generated by esbuild)
```

### Critical: Workers vs Node.js Boundary

**`src/lib/cloudflare.ts`** ŌĆö Workers-compatible. Imported by production server code. Contains API clients only (HTTP calls). **No Node.js built-ins allowed.**

**`src/lib/cloudflare-etl.ts`** ŌĆö Node-only. Imported by `scripts/*.ts` only. Contains CLI wrappers using `execSync`, `writeFileSync`, etc. **Never import this from production code paths** (src/index.ts, src/tools/, src/resources/, src/lib/bible-utils.ts).

Violating this boundary causes Cloudflare Workers deployment failure (error 10021: unsupported Node.js built-in).

### Framework Pattern

**Registration:** All capabilities (tools, resources, prompts) follow a decorator pattern:
1. Define handler function with specific signature
2. Attach metadata (`.description`, `.input`, `.mimeType`)
3. Register with server via `server.tool()`, `server.resource()`, or `server.prompt()`

**Handler signatures:**
- **ToolHandler:** `(args, ask?) => string | object | Promise<string | object>`
- **GeneratorToolHandler:** `function* (args) { yield progress; return result; }`
- **ResourceHandler:** `(params) => string`
- **PromptHandler:** `(args) => string | conversation(...)`

**Schema system:** The `T` namespace provides type-safe schema builders (`.string()`, `.number()`, `.boolean()`, etc.) with validation constraints.

### Tool Descriptions ŌĆö Dual Purpose Design

Tool descriptions (`.description` on each handler in `src/tools/*.ts`) serve two purposes:

1. **Usage documentation** ŌĆö What the tool does, its parameters, and output shape
2. **Client routing** ŌĆö Guide MCP clients toward complementary tools for comprehensive results

Every tool description should tell a client not just what THIS tool does, but what OTHER tools to use next. For example, `topical_search` points clients to `cross_references` (expand an anchor verse), `semantic_search` (additional verse-level discovery), and `word_study` (original-language depth). This creates a discoverable workflow graph that helps AI clients produce thorough Bible study sessions.

**When modifying tool descriptions:** Preserve cross-tool routing. Check the routing matrix ŌĆö every search-surface tool should point to at least one complement. Endpoint tools (`compare_translations`, `cross_references`) may omit outbound routing.

### Vectorize Indices

The server uses two Vectorize indices for semantic search:

**`bible-verses`** (155,510 vectors) ŌĆö Verse embeddings for semantic search across all verses. Index name is configurable via `VECTORIZE_INDEX_NAME` env var (empty string fallback).

**`bible-topics`** (5,385 vectors) ŌĆö Topic and theme embeddings for semantic topic matching:
- 5,319 vectors from Nave's Topical Bible categories (one per category)
- 66 vectors from book theme summaries (thematic summaries of each Bible book's major narrative arcs)

The topic index name is configurable via `BIBLE_TOPIC_INDEX_NAME` / `VECTORIZE_TOPIC_INDEX_NAME` environment variables.

Used by `topical_search` tool for multi-level thematic retrieval.

### Notable D1 Tables

**`nave_topic_book_salience`** ŌĆö Pre-computed per-book per-topic salience weights (relevance scores). Keys: `topic_id`, `book_number`. Contains normalized weights [0.0ŌĆō1.0] indicating how central each topic is to each Bible book. Enables efficient major witness ranking in topical search without requiring full topic vector computation at query time.

### Entry Point

`createServer()` initializes an MCP server instance. The returned `server.fetch` property is a Web Standard Fetch API handler compatible with:
- Cloudflare Workers
- Deno Deploy
- Node.js with adapters
- mctx hosting platform

The handler processes JSON-RPC 2.0 requests over HTTP.

### ETL Pipeline

Run ETL scripts locally (Node.js with `tsx`) to ingest data into Cloudflare D1 and Vectorize. Scripts are additive and must run in order:

**Phase 1 ŌĆö Core Data:**

**Phase 2 ŌĆö Search Infrastructure:**

**Phase 3 ŌĆö Topic Search (Additive):**
```bash

npm run search:topic-embeddings  # Ingest topic and book theme embeddings into Vectorize
```

**Convenience Scripts:**
```bash

npm run setup:infra         # Provision Cloudflare D1 and Vectorize infra via scripts/setup-infra.sh
```

All scripts use `CLOUDFLARE_ACCOUNT_ID` and `CLOUDFLARE_API_TOKEN` for local development.

## Environment Variables

### Cloudflare Credentials

Most server credentials follow a two-prefix fallback strategy: `BIBLE_*` for mctx deployment (primary) and a dev-specific name as fallback. Two variables ŌĆö `D1_DATABASE_ID` and `VECTORIZE_INDEX_NAME` ŌĆö do not follow this pattern; they use a single name in both environments (no `BIBLE_*` variant exists in the code).

| Variable | Purpose |
|---|---|
| `BIBLE_ACCOUNT_ID` | Cloudflare account ID ŌĆö mctx deployment (primary) |
| `CLOUDFLARE_ACCOUNT_ID` | Cloudflare account ID ŌĆö local dev fallback |
| `BIBLE_API_TOKEN` | Cloudflare API token ŌĆö mctx deployment (primary) |
| `CLOUDFLARE_API_TOKEN` | Cloudflare API token ŌĆö local dev fallback |
| `D1_DATABASE_ID` | D1 database ID ŌĆö single name, used in both mctx and local dev |
| `VECTORIZE_INDEX_NAME` | Vectorize verse index name ŌĆö single name, used in both mctx and local dev |
| `BIBLE_TOPIC_INDEX_NAME` | Vectorize topic index name ŌĆö mctx deployment (primary) |
| `VECTORIZE_TOPIC_INDEX_NAME` | Vectorize topic index name ŌĆö local dev fallback |

**`BIBLE_*` prefix** ŌĆö Used when the server is deployed on the mctx platform. Set these in the mctx dashboard under your server's environment configuration.

**`CLOUDFLARE_*` prefix** or other dev-specific prefixes ŌĆö Used for local development. Set these in your shell or `.env` file when running `npm run dev`.

The server always checks `BIBLE_*` first and falls back to alternatives, so both environments work without changing code. The `scripts/` directory (ETL scripts that run locally) uses dev-specific variable names and is unaffected by this convention.

## Testing Patterns

### JSON-RPC Helper Functions

**`createRequest(method, params)`** ŌĆö Creates a `Request` object with proper JSON-RPC 2.0 structure:
```typescript
{
  jsonrpc: '2.0',
  id: 1,
  method: 'tools/call',
  params: { name: 'find_text', arguments: { query: 'love your neighbor' } }
}
```

**`getResponse(response)`** ŌĆö Parses JSON-RPC response from `Response` object.

### Test Structure

Tests organized by capability type:
- **Tools** ŌĆö Invoke via `tools/call`, validate `result.content[0].text`
- **Resources** ŌĆö Read via `resources/read`, validate `result.contents[0].text` and `mimeType`
- **Prompts** ŌĆö Get via `prompts/get`, validate `result.messages[]` array structure
- **Server Capabilities** ŌĆö List via `tools/list`, `resources/list`, `prompts/list`

Error handling tests verify `result.isError === true` and error message content.

## GitHub Actions Security

**ALL GitHub Actions MUST be SHA-pinned. NO EXCEPTIONS.**

### The Rule

Reference actions by full commit SHA, never by tag or branch name.

**Why this matters:**
- **Tags are mutable** - Can be force-pushed to point to malicious code
- **Supply chain security** - Prevents compromised action maintainers from injecting code into your workflows
- **Reproducibility** - Guarantees exact same action code runs every time
- **Audit trail** - Clear record of exactly which version ran

### Examples

ŌØī **WRONG - Tag references:**
```yaml
uses: actions/checkout@v4
uses: actions/setup-node@v4
uses: docker/build-push-action@v5
```

Ō£ģ **CORRECT - SHA-pinned with version comment:**
```yaml
uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683 # v4.2.2
uses: actions/setup-node@39370e3970a6d050c480ffad4ff0ed4d3fdee5af # v4.1.0
uses: docker/build-push-action@4f58ea79222b3b9dc2c8bbdd6debcef730109a75 # v5.0.0
```

### Comment Convention

Always include a trailing comment with the version tag for human readability:
- Makes reviews easier (reviewers see version at a glance)
- Simplifies upgrades (know which tag to look for when updating)
- Documents intent (SHA alone is opaque)

**Format:** `uses: owner/repo@<full-sha> # <version-tag>`

### Finding SHAs

```bash
# Get SHA for a specific tag
gh api repos/actions/checkout/git/ref/tags/v4.2.2 --jq '.object.sha'

# Or browse releases on GitHub
# https://github.com/actions/checkout/releases
```

### Enforcement

All workflow files in `.github/workflows/` must comply. PRs with tag-based action references will be rejected.

## Branch & Deployment Architecture

### Two-Branch Model

| Branch | Purpose | Contents |
|--------|---------|----------|
| `main` | Source of truth | Full source code, scripts, tests, configs |
| `release` | Deployment artifact | `dist/index.js`, `package.json`, `README.md`, `.release-hash` only |

**`release` is NOT a source branch.** It contains only built artifacts synced by the release workflow. Do not manually edit files on `release`. The stale `src/index.ts` on `release` is historical baggage from branch initialization ŌĆö it is not part of the deployment path.

### Deployment Pipeline

1. Push to `main` triggers `.github/workflows/release.yml`
2. Workflow builds `dist/index.js` from `main` source via esbuild
3. If the dist hash changed, workflow syncs `dist/`, `package.json`, and `README.md` to `release`
4. mctx platform detects the `release` branch update and deploys

**Consequence:** All code fixes go to `main` only. Never cherry-pick source code changes to `release` ŌĆö the workflow handles artifact syncing. Cherry-picks will fail due to structural divergence (release has no `src/lib/`, `src/tools/`, `scripts/`, etc.).

### What Gets Synced to Release

The release workflow syncs these files from `main` to `release`:
- `dist/index.js` ŌĆö the production bundle
- `package.json` ŌĆö version, description, metadata (consumed by mctx platform)
- `README.md` ŌĆö displayed on the mctx server page
- `.release-hash` ŌĆö hash of the last synced dist, used to skip no-op releases

If any of these are not being synced, the release workflow needs to be updated.

## Public-Facing Changes ŌĆö README & package.json Review

Any change that affects external behavior, user-visible output, or tool capabilities **must** trigger a review of:

1. **`README.md`** ŌĆö This is the public product page on mctx.ai, not developer docs. Review against mctx's recommended 7-section structure (Opening Hook, Differentiation, What You Get, How It Works, Using This Server, Example Phrases, Real-World Workflows). See [mctx docs on writing a good README](https://docs.mctx.ai/building-mcp-servers/server-requirements#writing-a-good-readme).

2. **`package.json` `description` field** ŌĆö First 100 characters are the hook shown in registry listings. Must reflect current capabilities. See [mctx docs on description](https://docs.mctx.ai/configuration/mctx-json#description).

**Triggers:** New tools, changed tool behavior, new response fields, renamed capabilities, removed features. Internal refactors that don't change external behavior do NOT trigger this.

## Version Bump Process

**Admin-only operation.** Version bumps require branch protection bypass (admin privileges). Only Karl can push these.

### Steps

1. Update `version` in `package.json` on `main` (run `npm install` to sync `package-lock.json`)
2. Commit on `main`: `chore: bump version to X.Y.Z`
3. Push `main` ŌĆö the release workflow will sync `package.json` to `release` automatically

### Notes

- No code changes ŌĆö version bump only
- Requires admin bypass for branch protection on `main`
- Always wait for any in-flight deployments to finish before pushing
- Do NOT manually cherry-pick to `release` ŌĆö the release workflow handles syncing
